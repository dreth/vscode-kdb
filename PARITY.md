# Standalone parity and architecture

This document is an evidence ledger, not a parity declaration. `vscode-kdb` is a standalone first-party implementation, but the `0.1.5` code has **not** completed functional or visual parity verification with KDB-X, q Professional, or other broad q development environments.

The comparison baseline is:

- `vscode-kdb` `0.1.5`, including the focused first-party Server Explorer and Query History, unsaved-value connection testing, fixed original-domain chart reset, optional qText readability, and earlier standalone foundations.
- `kdb-sqltools` `0.3.17` at `af2c7c9`. Its repository was inspected read-only. References below name source and test symbols at that commit; pre-existing uncommitted generated-site changes in that checkout were not used as evidence.
- The connection reliability comparison also examined `kdb-sqltools` commit `f2aab16` (`fix: harden kdb connection reliability`).
- q Professional's public `jshinonome/vscode-k-pro` snapshot at `fc9afacaeaf5e90eb013eb34426488841cc24f2a`. Its formatter documentation informed product-level readability research only. The repository license is all-rights-reserved; no code, logic, or assets were copied.
- Apache-2.0 `KxSystems/kx-vscode` at `1c745bf0221dd3cca85dce925c4d432d80bb5ef5`. Its source exposes qlint integration as linting rather than a general qText pretty-printer. No code, logic, or assets were adapted for 0.1.5.

The executable cross-extension gate is documented in [`test/parity/README.md`](test/parity/README.md). Its checked [`PARITY_RUN.md`](PARITY_RUN.md) / [`PARITY_RUN.json`](PARITY_RUN.json) snapshot ran standalone commit `0c1ca7ddfb4eb48f788f8b32cf865340da2410dd` against the pinned reference and real q. It classified all 63 cases across 381 assertions: 49 `PASS`, 5 `DIFFERENT_BY_DESIGN`, 3 `GAP`, and 6 `NOT_TESTABLE_HERE`; the result is `VALID_WITH_KNOWN_GAPS`, not source-of-truth sign-off. Those executable classifications are bounded case outcomes. The broader Present/Partial/Missing labels below describe current product state and must not be inferred from one passing core case.

Status meanings:

- **Present**: the standalone path exists and has direct automated evidence appropriate to this pass.
- **Partial**: useful implementation exists, but behavior, coverage, UX, or a reference capability is still incomplete.
- **Missing**: the standalone path does not exist. Rows marked intentional must not be recreated merely to mimic SQLTools.

## Competitive capability audit

Server Explorer and Query History close two practical standalone workflow gaps, but their presence is not evidence of full KDB-X or q Professional parity. The explorer intentionally covers one active direct connection and one configured namespace, with read-only metadata and confirmed previews. It does not reproduce multi-environment administration, Insights/gateway workflows, server management, broad namespace navigation, write operations, notebooks, or every editor convenience. Query History is similarly focused on issued editor runs and conservative local privacy rather than becoming a session-management subsystem.

- [q Professional / `jshinonome/vscode-k-pro` at `fc9afacaeaf5e90eb013eb34426488841cc24f2a`](https://github.com/jshinonome/vscode-k-pro/tree/fc9afacaeaf5e90eb013eb34426488841cc24f2a) documents a formatter and was reviewed for product inspiration only. Its public repository is all-rights-reserved; no code, logic, or assets were copied.
- [KX's `KxSystems/kx-vscode` at `1c745bf0221dd3cca85dce925c4d432d80bb5ef5`](https://github.com/KxSystems/kx-vscode/tree/1c745bf0221dd3cca85dce925c4d432d80bb5ef5) was inspected under Apache-2.0. Its `kdb.ls.q.lint` command is qlint integration—linting, not a general qText result pretty-printer. No code, logic, or assets were adapted for 0.1.5, so no new Apache-derived file or third-party notice is claimed.
- SQLTools and `kdb-sqltools` remain absent as runtime/UI dependencies. The standalone views, storage, execution integration, and tests are KX-owned implementations rather than SQLTools adapters.

## Architecture boundary

No product/runtime monorepo, cross-repository dependency, symlink, generated copy, or SQLTools package was introduced. The package-excluded test harness loads each checkout's compiled core modules only for same-fixture comparison; no reference code enters the standalone runtime or artifact.

Later extraction should start **inside `vscode-kdb`**, preserving it as the future source-of-truth candidate:

| Future boundary | Safe core | Keep in the standalone adapter / compatibility adapter |
| --- | --- | --- |
| q wire and values | q IPC codecs, receive buffering, q value types, q error type | TCP socket lifecycle, OutputChannel wiring, SQLTools driver conversion |
| q execution | namespace wrappers and script grouping helpers | editor selection policy, commands, connection picker, SQLTools current-block aliases |
| results | columnar storage, cell formatting, sorting, text/XLSX serialization limits | VS Code webview lifecycle, clipboard/save dialogs, command IDs, setting namespaces |
| charting | inference, sampling, grouping, OHLC validation/aggregation, box statistics | webview controls, uPlot lifecycle, theme and persisted panel state |
| local data | token/range/format validation and response serialization | listener ownership, active-panel routing, notifications |
| diagnostics | allowlisted fields, secret redaction, duration formatting | `KX` OutputChannel and extension-specific lifecycle events |

The first extraction step is a file-boundary refactor with unchanged behavior and parity tests in this repository. Only after those interfaces stabilize should a compatible subset be deliberately backported to `kdb-sqltools`. A live dependency from standalone to SQLTools, or a shared SQLTools UI/session abstraction, is explicitly out of bounds.

## Milestones

### M0 — foundation (this pass)

- Track this matrix and the source-of-truth/backport direction.
- Add strict, reproducible MkDocs sources, committed output, normalization, and build-only Pages verification.
- Add the `KX` diagnostics channel, safe lifecycle/timing events, redaction tests, and direct q verification.
- Package `0.1.5` for direct user testing only; do not publish it to Marketplace.

### M1 — automated confidence

- Maintain and extend the proven cross-extension core gate without importing SQLTools into the product. Add standalone panel-protocol, search/sort, settings, guardrail, and webview behavior cases that remain outside the current pure-core fixtures.
- Add KX panel message-contract tests and an Extension Host smoke suite for activation, connection tree state, command execution, cancellation UI, and settings.
- Add general CI for compile, focused tests, Extension Host smoke, optional live q, documentation drift, and a non-publishing VSIX inventory/credential scan.
- Test simultaneous panels and queued work so local cancellation remains explicit and deterministic.

### M2 — first-party capability decisions

- Evaluate the focused Server Explorer and privacy-aware Query History with users before broadening either surface; preserve their independent opt-in controls and direct-q semantics.
- Decide whether q-block execution, secure remote profiles/SSH, auto-chart commands, and stronger server interruption belong in the first-party product; document security and q semantics before implementation.
- Do not expand the explorer with placeholder namespace, SSH, TLS, gateway, Insights, broker, administration, or write controls.

### M3 — experimental notebook evaluation

q notebooks are a later experiment, not a parity checkbox or a promised 0.1.x feature. Before contributing a notebook type or controller, define a durable file format and serializer, cell identity and execution-state restoration, stale/error/cancellation behavior across cells, renderer/controller and Extension Host test coverage, interoperability expectations, and the package-weight cost of VS Code notebook infrastructure. Do not ship a placeholder controller or silently reinterpret `.q` files as notebooks.

### M4 — functional and visual parity verification
