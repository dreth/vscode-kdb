# Standalone parity and architecture

This document is an evidence ledger, not a parity declaration. `vscode-kdb` is a standalone first-party implementation, but the `0.2.3` code has **not** completed functional or visual parity verification with KDB-X, q Professional, or other broad q development environments.

The comparison baseline is:

- `vscode-kdb` `0.2.3`, including the native `KX q (Direct IPC)` Jupyter NotebookController, live extension-host notebook result records plus bounded persisted MIME snapshots, the separate Python/IPython `%%q` helper route, actual q language mode for selected Jupyter code cells, the one-shot legacy KDB connection migration bridge, the focused first-party Server Explorer and Query History, unsaved-value connection testing, fixed original-domain chart reset, optional qText readability, and earlier standalone foundations.
- `kdb-sqltools` `0.3.17` at `af2c7c9`. Its repository was inspected read-only. References below name source and test symbols at that commit; pre-existing uncommitted generated-site changes in that checkout were not used as evidence.
- The connection reliability comparison also examined `kdb-sqltools` commit `f2aab16` (`fix: harden kdb connection reliability`).
- q Professional's public `jshinonome/vscode-k-pro` snapshot at `fc9afacaeaf5e90eb013eb34426488841cc24f2a`. Its formatter documentation informed product-level readability research only. The repository license is all-rights-reserved; no code, logic, or assets were copied.
- Apache-2.0 `KxSystems/kx-vscode` at `1c745bf0221dd3cca85dce925c4d432d80bb5ef5`. Its source exposes qlint integration as linting rather than a general qText pretty-printer. No code, logic, or assets were adapted for 0.2.3.

The executable cross-extension gate is documented in [`test/parity/README.md`](test/parity/README.md). Its checked [`PARITY_RUN.md`](PARITY_RUN.md) / [`PARITY_RUN.json`](PARITY_RUN.json) snapshot ran standalone commit `0c1ca7ddfb4eb48f788f8b32cf865340da2410dd` against the pinned reference and real q. It classified all 63 cases across 381 assertions: 49 `PASS`, 5 `DIFFERENT_BY_DESIGN`, 3 `GAP`, and 6 `NOT_TESTABLE_HERE`; the result is `VALID_WITH_KNOWN_GAPS`, not source-of-truth sign-off. Those executable classifications are bounded case outcomes. The broader Present/Partial/Missing labels below describe current product state and must not be inferred from one passing core case.

Status meanings:

- **Present**: the standalone path exists and has direct automated evidence appropriate to this pass.
- **Partial**: useful implementation exists, but behavior, coverage, UX, or a reference capability is still incomplete.
- **Missing**: the standalone path does not exist. Rows marked intentional must not be recreated merely to mimic SQLTools.

## Competitive capability audit

Server Explorer, Query History, native direct-q notebook execution, and portable notebook snapshots close practical standalone workflow gaps, but their presence is not evidence of full KDB-X or q Professional parity. The explorer intentionally covers one active direct connection and one configured namespace, with read-only metadata and confirmed previews. The notebook controller intentionally uses that same active Direct IPC session and provides no remote-kernel orchestration. The product does not reproduce multi-environment administration, Insights/gateway workflows, server management, broad namespace navigation, write operations, every persistent widget interaction, or every editor convenience. Query History is similarly focused on issued runs and conservative local privacy rather than becoming a session-management subsystem.

- [q Professional / `jshinonome/vscode-k-pro` at `fc9afacaeaf5e90eb013eb34426488841cc24f2a`](https://github.com/jshinonome/vscode-k-pro/tree/fc9afacaeaf5e90eb013eb34426488841cc24f2a) documents a formatter and was reviewed for product inspiration only. Its public repository is all-rights-reserved; no code, logic, or assets were copied.
- [KX's `KxSystems/kx-vscode` at `1c745bf0221dd3cca85dce925c4d432d80bb5ef5`](https://github.com/KxSystems/kx-vscode/tree/1c745bf0221dd3cca85dce925c4d432d80bb5ef5) was inspected under Apache-2.0. Its `kdb.ls.q.lint` command is qlint integration—linting, not a general qText result pretty-printer. No code, logic, or assets were adapted for 0.2.3, so no new Apache-derived file or third-party notice is claimed.
- SQLTools and `kdb-sqltools` remain absent as runtime/UI dependencies. The standalone views, storage, execution integration, migration review, and tests are KX-owned implementations rather than SQLTools adapters. Reading an existing VS Code setting on explicit command invocation does not transfer SQLTools runtime/session ownership.

## Architecture boundary

No product/runtime monorepo, cross-repository dependency, symlink, generated copy, private Jupyter integration, or SQLTools package was introduced. The migration bridge retained in 0.2.3 contains only independently implemented schema knowledge and a read-only VS Code configuration ingress; no reference code or asset enters the standalone runtime or artifact. The package-excluded historical parity harness loads each checkout's compiled core modules only for same-fixture comparison.

Later extraction should start **inside `vscode-kdb`**, preserving it as the future source-of-truth candidate:

| Future boundary | Safe core | Keep in the standalone adapter / compatibility adapter |
| --- | --- | --- |
| q wire and values | q IPC codecs, receive buffering, q value types, q error type | TCP socket lifecycle, OutputChannel wiring, SQLTools driver conversion |
| q execution | namespace wrappers and script grouping helpers | editor selection policy, commands, connection picker, SQLTools current-block aliases |
| results | columnar storage, cell formatting, sorting, text/XLSX serialization limits | VS Code webview lifecycle, clipboard/save dialogs, command IDs, setting namespaces |
| notebooks | versioned bounded MIME schema, typed portable cells, escaping/static fallbacks, chart specification, pure selected-cell/language/default/marker models | VS Code supported document-language setter, NotebookRenderer/messaging, IPython evaluator registration, cell commands/metadata, controller/live session ownership |
| charting | inference, sampling, grouping, OHLC validation/aggregation, box statistics | webview controls, uPlot lifecycle, theme and persisted panel state |
| local data | token/range/format validation and response serialization | listener ownership, active-panel routing, notifications |
| diagnostics | allowlisted fields, secret redaction, duration formatting | `KX` OutputChannel and extension-specific lifecycle events |

The first extraction step is a file-boundary refactor with unchanged behavior and parity tests in this repository. Only after those interfaces stabilize should a compatible subset be deliberately backported to `kdb-sqltools`. A live dependency from standalone to SQLTools, or a shared SQLTools UI/session abstraction, is explicitly out of bounds.

### M3 — bounded notebook renderer/protocol (0.2.0)

That release shipped a real NotebookRenderer, versioned bounded MIME serializer, durable `%%q`/namespaced metadata convention, static export fallbacks, and focused explicit-evaluator IPython helper. It did not contribute a placeholder controller or reinterpret `.q` files as notebooks. Renderer reopening worked from saved output without the originating live session; only the saved preview was available.

At that milestone, extension-driven same-session cell execution, live-result ownership, and controller-owned cancellation were deliberately deferred and disabled. The 0.2.3 direct controller closes that architecture gap through supported VS Code notebook APIs and the existing first-party connection lifecycle; private Jupyter interception and hidden second connections remain prohibited.

### M3.1 — actual notebook q language UX (0.2.2)

That release used VS Code's supported document-language setter to make selected Jupyter code-cell documents genuinely q, added discoverable toolbar/context/Command Palette actions, restored a safely resolved notebook default, and kept one durable `%%q` preparation path for the current Python-kernel evaluator. VS Code's built-in serializer records a non-default q cell under raw `metadata.vscode.languageId`; at that milestone the generic picker was controller-filtered and no q-capable controller existed.

This did not make the normal Python controller execute q-language cells. The Python `%%q` helper remains supported as a separate same-Python-kernel route in 0.2.3 and still does not share state with Direct IPC by implication.

### M3.2 — native KX q Direct IPC controller (0.2.3)

The release registers `vscode-kdb.q-notebook-controller` through `vscode.notebooks.createNotebookController` for `jupyter-notebook`, advertises q, activates on Jupyter notebooks, and appears in VS Code's normal kernel/controller selector as **KX q (Direct IPC)**. Normal notebook Run executes the complete q cell through the existing active KX connection, client, and namespace. It does not intercept the Python controller, call private Jupyter APIs, or create a per-cell/per-notebook connection.

Direct live results use the first-party KX result model and shared durable `vscode-kdb.results.*` settings while their in-memory records exist. Each opaque record is bound to its notebook URI and cell URI; rerun replaces it, cell removal removes it, notebook close removes it, deactivation clears all records, and the 512-record cap evicts the oldest. Saved `.ipynb` output contains only the safe bounded portable snapshot; reopening cannot recover omitted data. Provider/fake and contract tests cover these boundaries, but a visual or real Extension Host selector/run test remains outstanding.

### M5 — source-of-truth switch and backports

After user sign-off on M4, `vscode-kdb` becomes the sole forward-development source of truth:

1. Land and release the standalone fix or feature first, with its tests and documentation.
2. Open an explicit `kdb-sqltools` compatibility backport referencing the standalone commit/tag and listing adapter differences.
3. Backport only pure/core behavior that fits the SQLTools contract; keep SQLTools commands, driver types, SSH hookup, and legacy settings in that repository.
4. Run the full `kdb-sqltools` unit, Electron E2E, live-q, docs, and package gates before its compatible release.
5. Never forward-port a reference-only compatibility change automatically. Re-evaluate it as a first-party product proposal.

Until M4 is signed off, every release note and documentation page must describe the standalone phase and its known gaps rather than claim parity.
